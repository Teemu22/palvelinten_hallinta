# H5 - Gitar Hero

## Tiivistelmät

### What is Git?
- Git on versionhallintajärjestelmä, jonka avulla voidaan seurata tiedostoihin tehtyjä muutoksia ajan kuluessa
- Se mahdollistaa palaamisen aiempiin versioihin ja muutosten vertailun
- Git tallentaa projektista tilannekuvia (snapshot), ei pelkästään muutoksia
- Jokaisella käyttäjällä on oma paikallinen kopio projektista
- Muutokset voidaan jakaa muiden kanssa esimerkiksi GitHubin kautta

**Oma huomio: Git ei ole pelkkä tiedostojen siirtotyökalu, vaan sen pääidea on muutosten hallinta ja yhteistyön helpottaminen.**

### Gitin peruskomennot
- ```git add``` lisää muutetut tai uudet tiedostot versionhallintaan (staging area)
- ```git commit``` tallentaa muutokset ja luo uuden version projektista
- ```git push``` lähettää paikalliset muutokset etävarastoon (esim. GitHubiin)
- ```git pull``` hakee etävaraston muutokset paikalliseen projektiin
- ```git add``` valitsee, mitä muutoksia halutaan mukaan seuraavaan commitiin
- ```git commit``` vaatii viestin, jossa kuvataan mitä muutoksia on tehty
- ```git push``` tekee muutokset näkyväksi muille käyttäjille
- ```git pull``` varmistaa, että oma versio on ajan tasalla ennen uusien muutosten tekemistä

**Oma huomio: Näitä komentoja käytetään yleensä yhdessä järjestyksessä add → commit → push, ja pull ennen sitä, jotta vältetään ristiriidat.**

### c.) Doh!

Tässä tehtävässä kokeiltiin tilanteen peruuttamista Gitissä ilman commitointia. Tavoitteena oli tehdä muutos tiedostoon, lisätä se staging-alueelle ja tämän jälkeen peruuttaa kaikki muutokset git reset --hard -komennolla.

Tämä vastaa tehtävänannon vaatimusta, jossa huonot muutokset poistetaan ennen commitointia.

<img width="1279" height="235" alt="image" src="https://github.com/user-attachments/assets/2f9eaab0-4e08-4dd5-b9b6-10b4115d5c6f" /> **Kuvassa tiedostoon b_dolly.txt tehty muutos ennen versionhallintaan lisäämistä.**

Muokkasin olemassa olevaa tiedostoa b_dolly.txt ja lisäsin siihen uuden rivin. Tämä simuloi tilannetta, jossa tiedostoon tehdään virheellinen tai ei-toivottu muutos.

<img width="1281" height="301" alt="image" src="https://github.com/user-attachments/assets/231123df-0fc5-4375-bbbc-fe973e465acb" /> **Kuvassa muutos lisättynä staging-alueelle git add -komennolla, mutta sitä ei ole vielä commitoitu.**

Seuraavaksi lisäsin muutoksen staging-alueelle git add --all -komennolla. Tässä vaiheessa muutos on valmis commitointia varten, mutta sitä ei ole vielä tallennettu versionhallinnan historiaan.

<img width="1281" height="187" alt="image" src="https://github.com/user-attachments/assets/067a4cf5-e281-4bd3-b797-bec5e4aae4fb" /> **Kuvassa git reset --hard -komennon jälkeen ei ole muutoksia, eli kaikki tekemättömät muutokset on poistettu.**

Tämän jälkeen peruutin kaikki tekemättömät muutokset git reset --hard -komennolla. Tämä komento palauttaa projektin viimeisimpään commit-tilaan ja poistaa kaikki staging-alueella ja työtilassa olevat muutokset.

<img width="1285" height="174" alt="image" src="https://github.com/user-attachments/assets/50937ef8-950e-46f7-a828-8cd6d6197a72" /> **Kuvassa tiedoston sisältö resetoinnin jälkeen. Aiemmin tehty muutos on positunut.**

Lopuksi tarkistin tiedoston sisällön ja huomasin, että aiemmin tehty muutos oli poistunut. Tämä vahvistaa, että git reset --hard toimii tehokkaana tapana peruuttaa ei-toivotut muutokset ennen commitointia.

On tärkeää huomata, että tämä komento poistaa muutokset pysyvästi, eikä niitä voi palauttaa helposti.

### d.) Tukki

Tässä tehtävässä tarkastelin Gitin commit-historiaa ja analysoin, mitä muutoksia tiedostoihin on tehty. Tavoitteena oli ymmärtää, miten Git tallentaa muutokset sekä varmistaa, että käyttäjätiedot (nimi ja sähköposti) näkyvät oikein commit-historiassa.

Tämä vastaa tehtävänannon vaatimusta, jossa tarkastellaan repositoryn historiaa ja tehtyjä muutoksia.

<img width="1289" height="466" alt="image" src="https://github.com/user-attachments/assets/3877e30b-0698-4cfb-ad2b-2f581c1891de" /> **Kuvassa Git commit-historia git log -komennolla, jossa näkyvät tehdyt muutokset ja tekijän tiedot.**

Aloitin tarkastelemalla commit-historiaa git log -komennolla. Komento näyttää kaikki tehdyt commitit aikajärjestyksessä sekä niihin liittyvät tiedot, kuten tekijän nimen, sähköpostin ja commit-viestin.

<img width="1272" height="798" alt="image" src="https://github.com/user-attachments/assets/37e8f30c-bdd0-444f-b3fd-88d31242c213" /> **Kuvassa git log --patch -komento, joka näyttää tarkasti mitä muutoksia tiedostoihin on tehty.**

Seuraavaksi käytin git log --patch -komentoa, joka näyttää tarkemmin, mitä muutoksia tiedostoihin on tehty. Komennon avulla voidaan nähdä lisätyt (+) ja poistetut (-) rivit, mikä auttaa ymmärtämään muutosten sisällön tarkasti.

<img width="1286" height="211" alt="image" src="https://github.com/user-attachments/assets/9e1b816e-a5c3-4fa6-a8dc-48e567a3cd14" /> **Kuvassa Gitin käyttäjätiedot (nimi ja sähköposti), jotka näkyvät commit-historiassa.**

Lopuksi tarkistin Gitin käyttäjätiedot git config -komennolla. Näiden tietojen perusteella commit-historiassa näkyy, kuka on tehnyt muutokset.

Tässä tapauksessa nimi ja sähköposti olivat oikein, joten niitä ei tarvinnut muuttaa.

Tämän vaiheen perusteella voidaan todeta, että Gitin loki tarjoaa selkeän näkymän projektin muutoksiin ja niiden tekijöihin. Tämä on tärkeää versionhallinnassa, sillä sen avulla voidaan seurata projektin kehitystä ja ymmärtää tehtyjä muutoksia.
